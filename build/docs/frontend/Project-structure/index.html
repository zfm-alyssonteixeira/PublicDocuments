Project-structure

General​

  • Communication within a layer is allowed
  • Communication between different layers is only allowed as drawn in the figure above
  • The system is divided into different components, such as ContactManagement or OrganizationManagement

Components​

  • There must be no communication between the components
  • Each component should be interchangeable
  • Within each component a separate layered architecture is implemented

Layers

(from outside to inside)

Route​

  • Validates if model id is an integer
  • Checks

LaravelData (Package)​

  • Structures incoming request data to a data object
  • Validates types of incoming request data
  • Deletes not listed properties

Controller​

  • Receives the validated LaravelData request objects
  • Can either be a ...ManagementController or an invokable Controller
  • Does not contain any logic
  • Calls service to process data
  • Sends the response as a LaravelData DTO

Service​

  • Contains the processing logic
  • Has no dependencies to controller or functions of Laravel
  • Performs interactions to functionalities of Laravel via repositories
  • Contains at the end only imports from repositories or from services (of this component)
  • Has no possibility to call methods on models or on model objects

Repository​

  • Performs all requests to models
  • Calls functionality from Laravel
  • Model functions (e.g. User::all()) are called in the repository
  • Calls functions on models, such as $project->contacts()->get().

Model​

  • Specifies data structures for DB
  • Includes all specific methods that are common for Laravel in the model, such as getFullName or relation methods
